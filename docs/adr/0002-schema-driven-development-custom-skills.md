# ADR 0002: スキーマ駆動開発プロセスのカスタムスキル定義と適用

## ステータス
承認済み (Accepted)

## コンテキスト
本プロジェクトでは、スキーマ駆動開発 (Schema-Driven Development) を標準プロセスとして採用しています。
OpenAPIから始まり、ent schema、Atlas Migration、そしてデータベース物理スキーマへと流れる一方向の Single Source Of Truth (SSOT) の整合性を強制し、AIエージェントによる自動生成コードの誤った修正や手書きマイグレーションなどのアンチパターンを防止するため、AIエージェントのカスタムスキル（`.claude/skills`）を整備・更新する必要がありました。
また、データベース性能や同時実行性を維持するため、トランザクションの適切な制御（排他制御、外部API/sleep排除、リトライ、バッチ化）に関する指針を定義し、各設計・実装レイヤー間で連携させる必要がありました。
さらに、本番/ステージング環境のスロークエリやロック競合を検知した際に、EXPLAIN、トレース、およびpprofを活用して自動的にスキーマやコードの修正案を導き出せるよう、専門のプロファイリングスキルを追加する必要がありました。

## 意思決定
以下の通りカスタムスキル定義を更新・新規作成しました。

1. **`database-design` スキルの更新**
   - **目的**: 業務要件からDB設計・ent schema・Migrationを生成するプロセスの自動化・標準化。
   - **トランザクション分析の導入**: 7つの考慮事項（トランザクション境界、ロック範囲、分離レベル等）と5つの実施原則（短時間化、外部API/sleep禁止等）を規定し、スキーマ見直しやコード修正へ連携。
   - **出力順**: ドメイン分析、エンティティ一覧、ER図、ent schema、Goコード、Atlas migration、SQL、インデックス提案、性能上の懸念、リスク。

2. **`openapi-design` スキルの更新（新規作成）**
   - **目的**: API要件からOpenAPI定義を生成・設計するプロセスの標準化。
   - **トランザクション境界の考慮**: 長時間ロックを回避するため、外部通信を含む同期処理を避けて非同期受付パターンやAPI分割の改善提案を盛り込むAPI設計方針を規定。
   - **出力順**: API設計方針、OpenAPI(YAML形式)、改善提案。

3. **`golang-design` スキルへの `schema-sync` およびトランザクション実装ルールの統合**
   - **目的**: OpenAPI, ent schema, DB schema間のスキーマ整合性監査、およびトランザクションGoコードの最適化。
   - **確認項目**: フィールド不足、型不一致、Required不一致、Nullable不一致、Enum不一致、命名不一致、Index不足。
   - **トランザクション最適化**: 外部API/sleep排除のためのGoコードリファクタリング、バッチ挿入、Serializable等でのエラー時の指数バックオフ・リトライ処理の実装指針を追加。

4. **`transaction-profiling-analyzer` スキルの新規作成**
   - **目的**: EXPLAIN、トレース、pprofを横断したトランザクション・ボトルネックの自動プロファイリング。
   - **分析対象**: Seq Scanの検知、OTelを用いたTx生存時間・ブロッキング処理の特定、pprofによるロック（mutex/block）プロファイリング、およびデッドロック Lock Graph 分析。
   - **出力形式**: 分析サマリー、スキーマ見直し提案（database-designと連携）、Goコード修正案（golang-designと連携）。

5. **`golang-observability` スキルの更新**
   - **目的**: トランザクション・プロファイリングに必要な計装の標準化。
   - **変更点**: スロークエリやエラーの slog 属性規約、OTel によるDBトランザクションセマンティック属性および親スパンの計装規約、pprof における mutex/block サンプリングレートの設定規約を追加。

6. **共通制約の強制適用**
   - すべての関連スキルにおいて、以下のルールをヘッダーの共通ポリシーとして強制：
     - **SSOT**: OpenAPI → ent → Atlas Migration → Database
     - **禁止事項**: 生成済みファイル直接編集禁止、OpenAPIとent同時編集禁止、migration手書き禁止
     - **命名規則**: API: camelCase, DB: snake_case, Go: PascalCase
     - **DBコアルール**: UUID利用、FK必須、N+1検出必須、soft delete制限

## 影響・結果
- AIエージェントが設計やスキーマ変更に関わるタスクを行う際、上記のルールが厳格に適用され、スキーマの不整合や不適切な実装が未然に防がれます。
- トランザクション境界がAPI・DB・実装の各レイヤーで一貫して考慮され、ロック競合や接続プールの枯渇を低減する堅牢な実装が実現します。
- スロークエリやデッドロック発生時に、計装されたログ・トレース・プロファイル（pprof）をもとに、AIエージェント自身が自律的にボトルネックを分析し、最適なスキーマおよびコードの修正提案（diff）を提示できるようになります。
