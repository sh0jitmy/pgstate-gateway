# 組織カスタムスキル：他リポジトリ展開ガイドおよび初期導入テンプレート
(Customer Support & Deployment Guide)

本ドキュメントは、本プロジェクトで構築した「組織カスタムスキルおよび監査・検査ガバナンスの仕組み」を、他のGoリポジトリや新規プロジェクトへスムーズに横展開するための手順書およびテンプレート集です。

カスタマーサポート（Customer Support）の視点に基づき、開発に慣れていない初心者の方でもつまずかずに導入・運用できるよう、やさしい日本語で丁寧に解説しています。

---

## 🛠️ 初心者向けトラブルシューティング＆FAQ (カスタマーサポート窓口)

導入中に疑問が生じた場合は、まずこちらをご確認ください。

* **Q1. `make install` を実行した後に、どのようにスキルを使用すればよいですか？**
  * **回答**: `make install` を実行すると、カスタムスキルがグローバルディレクトリ（`~/.claude/skills/`）にコピーされます。これにより、PC上のすべての Claude Code で本スキルが自動的にロードされるようになります。特別な設定は不要です。

* **Q2. `make check` で文法エラー（Frontmatter validation failed 等）が発生します。どうすればよいですか？**
  * **回答**: 各スキルフォルダの `SKILL.md` の冒頭にある YAML フロントマター（`---` で囲まれた部分）に必要なキー（`name`、`description`、`user-invocable`、`metadata`、`allowed-tools`）がすべて正しく定義されているか確認してください。特にインデントやコロンの後のスペースが不足している場合にエラーが発生しやすくなります。

* **Q3. 既存のプロジェクトに導入したいのですが、Goのバージョンが異なります**
  * **回答**: スキル群は Go 1.18 以上であれば問題なく動作します。`.github/workflows/ci.yml` などの Go バージョン（例: `1.25` や `1.26`）を、お使いの環境に合わせて書き換えてご利用ください。

## 🚀 他リポジトリへの展開手順 (3ステップ)

他のプロジェクトへ本ガバナンスとスキルを導入する手順は、以下の3ステップのみで完了します。

### ステップ 1: スキル定義ファイルとチェッカーのコピー
新しいリポジトリのルートディレクトリに、スキル定義とスクリプトフォルダをそのままコピーします。

```bash
# 1. 新しいリポジトリへスキルとスクリプトをコピー
cp -R /path/to/claude_code_skills/.claude <新リポジトリのルート>/
cp -R /path/to/claude_code_skills/scripts <新リポジトリのルート>/
cp /path/to/claude_code_skills/Makefile <新リポジトリのルート>/
```

### ステップ 2: グローバル環境へのインストールと文法チェック
コピーしたスキルを動作環境の Claude Code に認識させ、文法チェックを行います。

```bash
# 2. グローバルディレクトリ (~/.claude/skills/) へインストール
make install

# 3. 各スキルファイルの文法チェックを実行
make check
```
「`🎉 All custom skills passed checks successfully!`」と表示されればステップ2は成功です。

### ステップ 3: 自己評価チェックリストの初期化
新しいリポジトリでの要件リストを作成し、初期同期を行います。

1. 後述の **「初期REQUIREMENTS.md テンプレート」** の内容で、新リポジトリのルートに `REQUIREMENTS.md` ファイルを作成します。
2. 以下のコマンドを実行して、適合率を初期化（0.00%）または現状に同期させます。
   ```bash
   make self-eval
   ```

- [ ] **R-2.1 セキュアロギング**: `main.go` にて `slog` のマスキング処理が実装されており、ログへの機密情報漏洩が防止されていること。

- [ ] **R-2.4 テストとビルドの保証**: `make test` および `make build` が正常にパスすること。
- [ ] **R-2.5 自動リリースの統合**: `.tagpr` および `.goreleaser.yaml`、各種GitHub Actionsワークフローが配置され、リリースフローが定義されていること。

- [ ] **R-2.10 OpenAPI 駆動開発の準拠**: `api/openapi.yaml` から `oapi-codegen` でコードを自動生成し、CIでの差分チェックが正常に通ること。
- [ ] **R-2.11 データベースアクセス (ent + SQLite)**: CGOを必要としない純粋な Go 実装の SQLite ドライバーを使用して、`ent` ORM 経由でデータベースアクセスおよび自動マイグレーションが動作すること。
- [ ] **R-2.12 Bearer 認証ミドルウェア**: `Authorization: Bearer` トークンを検証する Gin ミドルウェアが実装され、E2Eテストで検証されていること。

### ② 初期 ADR テンプレート (`docs/adr/0001-initial-setup.md`)
```markdown
# ADR 0001: 新規リポジトリへの組織カスタムスキル導入

## ステータス
提案中 (Proposed) / 承認済み (Accepted)

## コンテキスト
組織の品質基準、Go設計ベストプラクティス、およびセキュリティガバナンスを本リポジトリで統一して遵守するために、組織カスタムスキルおよび自己評価（Evals）の仕組みを導入する必要があります。

## 意思決定
以下の仕組みをリポジトリへ組み込み、エージェントによる開発ライフサイクル全体で適合検証を行います。
1. `.claude/skills/` による個別スキルの定義。
2. `make self-eval` による要件チェックリストの自動同期。
3. `make test` による単体・E2Eテストの自動化。

## 結果
- スキルに準拠したセキュアで信頼性の高いGoアプリケーションおよびインフラ設定が自動検証可能になります。
- 品質検査官による `inspection_report.md` の生成により、リリースごとの網羅性が客観的に証明されます。
```
