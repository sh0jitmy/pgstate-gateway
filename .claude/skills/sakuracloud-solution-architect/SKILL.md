# さくらのクラウド アーキテクチャ設計ガイドライン

## 1. コンポーネント選定と構成原則

さくらのクラウドが提供する多様なアプライアンスやリソースを適切に選択し、スケーラブルかつ運用負荷の低い構成を設計します。

- **コンピューティング**:
  - **仮想サーバ**: CPU/メモリの組み合わせが豊富。負荷に応じて動的にスケールアップ可能だが、ディスクのデタッチ/アタッチが必要な場合があるため、無停止スケールにはロードバランサを組み合わせたスケールアウトを優先する。
  - **サーバグループ（ホスト分散）**: 可用性が求められる冗長化サーバ（例: Webサーバ複数台）は、物理ホストの重複を避けるために「グループ分け」設定（同一ホストへの収容禁止）を必ず適用する。
- **ストレージ・ディスク**:
  - OS領域とデータ領域の分離（ディスクを複数アタッチ）。
  - 高速I/Oが必要なDB等は「SSD」を選択し、読み込み専用の一時領域やログ保存領域等には「HDD」を選択してコストを最適化する。
  - **オブジェクトストレージ（最新機能）**:
    - アーカイブデータや静的コンテンツの配信先として活用。
    - 2026年1月提供の**「リージョン間レプリケーション機能（石狩第1〜東京第1サイト間）」**を活用し、地理的に分散したDR（災害復旧）および遠隔地バックアップ構成を自動化設計する。
- **データベース（DBアプライアンス）**:
  - PostgreSQL / MySQL / Oracle などのマネージドデータベース。
  - 本番環境では、マルチAZ（ゾーン）に相当する冗長化構成（アクティブ/スタンバイ）を選択し、自動フェイルオーバーを有効化する。
- **インメモリキャッシュ（Redisアプライアンス）**:
  - セッション管理や一時キャッシュとして活用し、アプリケーションサーバのステートレス化を実現する。

## 2. ネットワークおよびセキュリティ設計

さくらのクラウドのネットワーク機能は、L2接続の柔軟性が高いことが特徴です。これを利用して堅牢なネットワークトポロジーを構築します。

- **VPCルータ（仮想ルータ）**:
  - 内部セグメント（プライベートネットワーク）と外部インターネットの境界に配置。
  - ファイアウォール、NAT（Source NAT / Destination NAT）、L2TP/IPsecによるリモートアクセスVPN、IPsecによる拠点間VPNを設計する。
- **セグメント分離とローカルスイッチ**:
  - インターネットから直接アクセスされるWeb/APIサーバは外部セグメント（またはロードバランサ経由）に配置。
  - DBアプライアンスやRedisアプライアンス、バックエンドAPIは、外部IPを持たない「スイッチ（ローカルスイッチ）」で構築されたプライベートセグメントに隔離する。
- **エンハンスドロードバランサ（ELB）（最新仕様）**:
  - コネクション数のメトリクス監視（モニタリングスイート連携）を活用し、システム全体のキャパシティプランニングとアラート設計を行う。
  - 実サーバとの接続やKeepaliveタイムアウト等の最新の接続仕様変更に適合するよう、バックエンドサーバ側のWebサーバ設定（Keepaliveタイムアウト値など）を整合させる。
- **WAF（Web Application Firewall）の選定**:
  - 外部公開Webサービスに対し、以下の特徴を踏まえてWAFを選定する：
    - **AIWAF-VE for さくらのクラウド**: 仮想アプライアンス型。柔軟な設定と国内DC内での高速なトラフィック処理。
    - **SiteGuard Server Edition**: ホスト型WAF。サーバにモジュールとして組み込み、低レイテンシで保護。
    - **攻撃遮断くん**: クラウド型WAF（新プラン体系）。シグネチャ自動更新による容易な運用管理。
- **パケットフィルタ**:
  - 各サーバのNICレベルで動作するフィルタ。不要なポートはインバウンド/アウトバウンドともに原則閉鎖し、最小権限（Least Privilege）を適用する。
- **ガバメントクラウドへの準拠（2026年3月採択）**:
  - さくらのクラウドがデジタル庁のガバメントクラウドとして正式に採択されたことに伴い、政府統一基準やISMAP（政府情報システムのためのセキュリティ評価制度）などの高水準なセキュリティ要件に準拠するよう、IAM、操作ログ監査、データ暗号化の設計を行う。

## 3. コストの見積もりと最適化（FinOps）

さくらのクラウド最大の強みである「インターネット転送料が無料」という特長を活かし、TCO（総所有コスト）を最小化します。

- **トラフィックフリーの最大活用**:
  - 大容量のメディア配信、ログ同期、バックアップ転送など、メガクラウドで大きなボトルネックとなる「Egress料金（データ転送量課金）」が発生するワークロードをさくらのクラウドに集約する。
  - メガクラウド（AWS/Azure/GCP等）とさくらのクラウドを専用線（ハイブリッド接続・ブリッジ接続）で結び、高負荷データ配信・処理部分をさくらのクラウドで引き受けることで、システム全体のネットワーク転送コストを大幅に削減する。
- **リソースのライトサイジング**:
  - 仮想サーバのCPUとメモリ比率をワークロード特性（メモリ集約型、CPU集約型など）に合わせて適切に選定し、過剰スペックを避ける。
  - 読み書き頻度の低いデータは高価なSSDではなく、HDDディスクやオブジェクトストレージへ段階的に移動（ライフサイクル管理）する。
- **課金体系の最適利用**:
  - さくらのクラウドの「日割」「月割」課金モデルの仕様を理解し、開発・検証環境など定時外に稼働が不要なサーバについては、スクリプト（USACLOUD等）で夜間・休日に自動シャットダウン・起動をスケジュールし、無駄な従量料金を抑制する。

## 4. Kubernetesおよびコンテナ実行環境の設計

要求仕様において Kubernetes (k8s) またはコンテナ型のプラットフォーム構築が求められている場合、さくらのクラウドの特性（マネージドk8s非提供）に基づき、以下のアーキテクチャを設計します。

- **Kubernetes (k8s) クラスタのセルフホスト設計**:
  - 仮想サーバ（または高火力PHY/VRT等）をノードリソースとし、`RKE2` や `kubeadm` を用いてコントロールプレーンおよびワーカーノードを自前構築するトポロジーを策定する。
  - 物理故障によるクラスタ全損を避けるため、各ノードのサーバには必ず「グループ分け（ホスト分散）」設定を有効化する。
- **クラウド基盤連携 (Cloud Controller Manager)**:
  - クラスタ内に **`sakura-cloud-controller-manager` (CCM)** を導入し、Kubernetes上の `Service (type: LoadBalancer)` の作成と、さくらのクラウドのマネージドロードバランサアプライアンスのプロビジョニング・ヘルスチェック設定を自動連携させる。
- **永続ストレージ (CSI) の代替設計**:
  - 公式のマネージドCSIドライバがないため、クラスタ内部でノード内ディスクを束ねて仮想分散ストレージを構成する **`Longhorn`** を導入するか、さくらのクラウドの「NFSアプライアンス」をマウントする構成を設計し、StatefulSet 等の永続化要件を満たす。
- **コンテナレジストリの活用**:
  - コンテナイメージのセキュアな保管・配置のため、さくらのクラウドが提供する「コンテナレジストリ」サービスを活用し、イメージの脆弱性スキャンやアクセス制御を実装する。
- **マネージドコンテナ「AppRun」の代替検討**:
  - k8sそのものの管理・保守（バージョン追従やクラスタ自体のレジリエンス運用）コストを回避したい場合は、Kubernetes/Knativeベースのサーバーレスコンテナ実行環境である **「AppRun」** を代替アーキテクチャとして検討し、運用負荷（SRE工数）を削減する。

## 5. AIおよびGPUコンピューティング設計

さくらインターネットが提供する強力なGPUリソースとAI APIを活用し、セキュアなAIインフラを構築します。

- **さくらのAI Engine**:
  - 生成AIアプリケーション（Chat completions, Embeddings, 音声処理）を素早く組み込むためのAPI推論基盤。
  - すべてのデータおよび処理が日本国内のデータセンターで完結するため、機密性の高いテキストや音声データのAI処理に適した構成を設計する。
- **高火力シリーズの選定**:
  - **高火力 DOK（コンテナ型）**: 機械学習の実験や一時的なトレーニングなど、コンテナ単位で手軽にGPUを利用するユースケース。
  - **高火力 VRT（VM型）**: 開発・テスト環境や、柔軟なリソース変更が必要なGPU推論・学習環境。
  - **高火力 PHY（ベアメタル型）**: NVIDIA H100/H200等を搭載した最高性能の専用物理GPUサーバ。大規模言語モデル（LLM）の本格的なファインチューニングや大量の推論処理を実行するインフラ。
- **機密AI (Confidential Computing)**:
  - 「高火力 PHY」などの環境で、Intel TDX（Trust Domain Extensions）や NVIDIA Confidential Computing (NCC) を有効化した設計を行う。
  - これにより、データセンター運用者や第三者からデータを完全に秘匿した安全な状態（Confidential VM/Baremetal）で、高度なデータ分析やAI推論（金融、医療、個人情報等）を実行するセキュリティ要件に対応する。

## 6. SRE・AIエンジニアとの役割連携

AI/GPUインフラや最新ネットワークの設計・運用を円滑に進めるため、SREチームおよびAIエンジニアと連携したアーキテクチャ設計を行います。

- **ソリューションアーキテクト（SA）の役割**:
  - 全体のインフラトポロジー、ネットワーク設計、セキュリティ境界（WAF/VPCルータ）、Kubernetes構成方式（CCM/CSI等）、コスト見積もりおよびFinOps最適化方針の決定。
- **AIエンジニアとの連携**:
  - **分担**: AIエンジニアがAIモデル選定（Qwen3-Coderや日本語特化オープンモデルなど）、Playgroundでのパラメータ調整、および「さくらのAI Engine」APIとのアプリケーション側接続設計を担当。
  - **連携**: SAはAI EngineのAPI利用制限やレイテンシの非機能要件、セキュリティ境界の安全性をAIエンジニアにフィードバックする。
- **SREチームとの連携**:
  - **分担**: SREが「高火力シリーズ」や「さくらのクラウド」のKubernetes/コンテナ環境（DOK等）のプロビジョニング、スケーリング自動化、CI/CDラインの構築を担当。
  - **連携**: SAは、Confidential Computingにおける「リモート構成証明 (Attestation)」の手順や鍵管理サーバー（KMS）の配置トポロジー、ログの監査証跡についてSREと設計をすり合わせる。

## 7. 自動化とインフラのコード化 (IaC)

### Terraform v3 (`terraform-provider-sakura`) 前提の設計

さくらのクラウドにおける自動化・構成管理には、最新のTerraformプロバイダ **`terraform-provider-sakura` (v3)** を前提とします。旧バージョンの `terraform-provider-sakuracloud` (v2) とは互換性がないため、以下の設計ガイドラインを遵守してください。

- **プロバイダの宣言**:
  最新の Terraform Plugin Framework を採用した `sacloud/sakura` プロバイダを明示的に指定します。
  ```hcl
  terraform {
    required_providers {
      sakuracloud = {
        source  = "sacloud/sakura"
        version = "~> 3.0"
      }
    }
  }
  ```
- **`filter` ブロック廃止への対応**:
  - v2で使われていた `filter` ブロック（各種リソースのインライン検索）はv3で廃止されています。
  - リソースの検索は、対応するデータソース（例: `sakuracloud_archive` や `sakuracloud_switch`）を使用するか、検索対象を特定できる属性値（IDやタグ）を直接変数として渡す設計に統一します。
- **型定義の厳密化への適応**:
  - v3では値の有無（Null/Empty）やリスト、マップの型定義が厳密化されています。
  - 属性値の参照時には、`can()` や `try()`、また明示的なデフォルト値（`coalesce()` 等）を用いて、想定外のNullエラーでプロビジョニングが失敗するのを防ぎます。

### USACLOUD (CLI) の活用

- `usacloud` CLIによる自動運用スクリプト（夜間のサーバシャットダウン/朝の起動など）を作成し、サーバ停止中のディスク料金のみに課金を抑える FinOps 自動化の仕組みを定義します。
