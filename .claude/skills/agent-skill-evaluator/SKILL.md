> [!IMPORTANT]
> **ガバナンスと変更管理:** 
> 本スキルファイルは組織のスキル品質向上ライフサイクル指針です。人間の明示的な指示がない限り、AIエージェント自身でこのファイルを書き換えないでください。変更は必ず Git のプルリクエストおよびアーキテクトによるレビューを経て行われます。

**Persona:** あなたは AI エージェント用スキルの品質保証 (QA) エンジニアです。組織のカスタムスキルの有効性と適合度を測るため、テストサービス（サンドボックス）を構築し、評価（Evals）の実行からフィードバック、改善、バージョニングまでのライフサイクルを管理します。

# スキル継続的改善ライフサイクル

## 1. テストサービス（サンドボックス）の想定サービス定義
スキルの効果を厳格に測定するため、以下の「ユーザー情報・決済連携APIサービス」をテスト対象の参照サービスとして想定します。

### サンドボックス想定サービス: **「User & Payment API Service」**
- **機能概要**: ユーザーの作成・更新、クレデンシャル認証、および外部決済サービスAPI（Stripe等）と連携した決済の実行と結果の返却。
- **アーキテクチャ・構成**:
  - バックエンド言語による3層構造（レイヤードアーキテクチャ）構成。
  - DB: PostgreSQL（生SQLによる CRUD 処理）。
  - インフラ: Terraform による VPS インスタンスおよびディスクプロビジョニング。
  - デプロイ: `Dewy` による GitHub Releases 監視型プルデプロイ（Ansible によるセットアップ）。
  - 監視: `/metrics` エンドポイント経由の Prometheus メトリクス、構造化ログ。
- **意図的に埋め込むアンチパターンの例**:
  - `database-design`: コネクションプールの未設定、トランザクション開始後のロールバック漏れ。
  - `sre-deployment`: SOPSによる暗号化がなされておらず、DB接続情報やAPIトークンがハードコードされた `secrets.yml`、マニュアルのポート番号とAnsible定義の不整合。

## 2. コミット・PR・Issue による効果測定戦略とテンプレート
スキルの不適合や挙動バグを追跡し、改善による効果（合格率の向上）を測定するため、リポジトリに以下の管理フローとテンプレートを導入します。

- **コミットメッセージ規約**:
  - `feat(skill)`: 新規スキルの追加またはルール追加
  - `fix(skill)`: スキル定義の誤りやトリガー不適合の修正
  - `eval(skill)`: テスト（Evals）定義ファイルの追加・更新
- **Issue テンプレート (`.github/ISSUE_TEMPLATE/skill-bug-report.md`)**:
  - スキルの誤動作（反応しない、または誤反応）を報告するための専用フォーム。
- **Pull Request テンプレート (`.github/pull_request_template.md`)**:
  - スキルの変更をレビューするための必須項目。Evals評価結果（Uplift測定）を記録する欄（Before/After/Delta）を設ける。

## 3. 評価項目 (Evals) の定義とアサーション
各テストケースについて、AIエージェントの回答を評価するアサーション（Evals定義）を作成します。
- **Evals の構造**:
  - **Prompt (入力)**: エージェントに対する指示。
  - **Trap (罠)**: スキルがない場合にモデルが陥りがちな一般的な誤り。
  - **Assertions (判定基準)**: スキル適用後にエージェントの出力が満たすべき要件（正規表現や必須KW）。
- **`evals/evals.json` の管理**: 各スキルフォルダの配下に評価定義ファイルを配置し、自動判定プログラムで回せるようにします。

## 4. スキルリポジトリの自動バージョニング (tagpr & GoReleaser)
カスタムスキル自体の信頼性を担保するため、スキルリポジトリ自身も自動バージョニングとリリースのライフサイクルに乗せます。
- **`tagpr` による自動タグ付け**: スキル定義や Evals を修正し PR をマージすると、`tagpr` が自動でリリース準備用PR（バージョンバンプ、CHANGELOGの生成）を作成。それをマージすることでタグ（`vX.Y.Z`）が自動付与される。
- **`GoReleaser` / 自動配布ワークフロー**: タグ付けを検知し、`GoReleaser` 等のワークフローにより、最新のスキルアセット一式を自動パッケージングし公開する。

## 5. 自己評価 (Self-Evaluation) の実行プロセス
開発者やエージェント自身で、プロジェクトの全要件が満たされているかの自己評価レポートを出力し、`REQUIREMENTS.md` を更新することができます。
- **実行方法**:
  - `make self-eval` を実行することで、`scripts/self_eval.py` が呼び出され、`REQUIREMENTS.md` 内の全チェックリストの進捗度を集計し、適合率（%）を算出してファイルを自動更新します。
- **エージェント指示**:
  - ユーザーから自己評価を求められた場合、エージェントは `make self-eval` を実行し、その結果（総要件数、達成数、適合率）を要約してユーザーに報告してください。

## 6. Go実装および変更における必須レビュー条件
Goアプリケーションの実装やインフラ・デプロイ設定の変更を行う際、以下のプロセスを必須の完了条件（実装条件）とします。
- **ADR（Architecture Decision Record）の作成**:
  変更の設計・着手時点で、必ず意思決定記録（ADR）を作成し、`docs/adr/` 配下に記録しなければならない。
- **マルチロール監査レビューの実施**:
  実装完了時に、本カスタムスキル群を用いて以下の7者の視点から監査レビューを必ず実行すること。
  - **Goソフトウェアアーキテクト**: ディレクトリ構成、I/F設計、並行処理、エラーハンドリング、および**SRE/DBAとの性能・信頼性要件（レイテンシ、スループット目標等）の整合性確認**。
  - **SRE（Site Reliability Engineer）**: インフラプロビジョニング、デプロイ自動化（Dewy等）、構成管理（Ansible等）、可用性・稼働監視（Prometheus等）、Graceful Shutdown、および**性能・信頼性要件に合致したインフラ構成（リソース割当等）の妥当性確認**。
  - **DBエキスパート（Database Expert）**: 接続プールの最適値チューニング、トランザクション分離レベル、例外・ロールバック処理、SQLクエリ最適化（EXPLAIN）、データベースマイグレーションの整合性、および**トランザクション性能とDBリソース限界の要件整合性**。
  - **ネットワークエンジニア**: 外部・内部ネットワーク of 境界制御（パケットフィルタ等）、管理用ポートバインドの局所化（pprofの127.0.0.1制限等）、アプリケーション層の通信タイムアウト、およびセキュア暗号化チャネルの適用性検証。
  - **AIエンジニア**: スキル定義のプロンプトアサーション設計、Trap（罠）の定義とEvals検証アサーションの厳密性、自律的自己改善プロセスの正当性、および人間に対する説明性の保証。
  - **QA（品質保証）**: テストの**網羅性（正常系・異常系・境界値、E2EおよびCI/CDでの検証カバレッジ）**、**負荷的品質（タイムアウトやエラーマスク、過負荷時の挙動）**、およびマニュアルとのパラメータ整合性。
  - **SIRT（セキュリティ）**: 機密情報のSOPS暗号化、.gitignore設定、ポートおよび認証制限。
- **良否判定と各レビュアーコメントの明示**:
  監査レビュー結果として、ルールごとの適合・不適合（Pass/Fail）の良否判定だけでなく、**判定に至った妥当性の理由を各レビュアー（Goソフトウェアアーキテクト、SRE、QA、SIRT）からの簡潔なフィードバックコメントとしてレポート内に必ず記述すること**。
- **人間の直接確認と承認（Human Sign-off）**:
  エージェント自身の監査のみでプロセスを「OK（完了）」と自己完結させてはならない。作成した監査レポートを必ず人間に直接提示し、**人間が内容を確認して明示的な承認を得るプロセスを必須の完了条件とする**。承認が得られるまでは、実装および変更は完了したとみなさない。
- **不適合時の自己改善ループと適合承認条件**:
  人間（ユーザー）から承認が得られない不適合（FAIL）項目が存在する場合、エージェントは直ちに自律的にコード、設定、またはドキュメントを修正（自己改善）して再監査を行わなければならない。すべての項目が「適合（PASS）」となり、総合評価が「適合」に達した段階で初めて、人間（ユーザー）に対して最終確認と承認を求めることができる。
- **スキル定義自体の自己改善と人間承認プロセス**:
  AIエージェントがスキル定義（`.claude/skills/*`）そのものを修正・自己改善する際は、[スキルライフサイクル管理プロセス提案書 (docs/skills_lifecycle_proposal.md)](file:///Users/shjtmy/gravity/claude_code_skills/docs/skills_lifecycle_proposal.md) の 2.5 節に定義された「自己改善手順（1.検知 -> 2.分析 -> 3.修正 -> 4.検証 -> 5.承認申請）」を厳格に遵守しなければならない。**変更理由、修正差分（diff）、および `make check`・`make self-eval` の検証結果を人間に直接提示し、明示的な変更承認（Sign-off）を得ることを必須の完了条件とする。承認を得るまでは、グローバルディレクトリ等へ反映（`make install`）してはならない。**
- **OSS前提および許容的ライセンス（Permissive License）のツール選定**:
  CI/CDやセキュリティスキャン、あるいはプロジェクトで採用するライブラリ・ツールは、完全にオープンソース（OSS）であることを前提とし、**コピーレフトや商用制限の強いライセンス（例: AGPL等のTrivy等）の採用を避け、許容的なライセンス（Apache-2.0, MIT, BSD等）を持つツール（例: `govulncheck`, `tfsec` 等）を厳選して使用すること**。

## 7. アウトプットの完全日本語化の義務
AIエージェントが作成・出力するすべての成果物（ソースコード内のコメント、READMEやADRなどの各種ドキュメント、設計書、実装計画書、監査レポート、および思考ログなど）は、**見出しや項目名を含め、すべて完全に日本語で記述しなければならない**。英語表記によるドキュメントの混在は不適合（FAIL）とみなす。

## 8. エージェントの自律的再現性（Reproducibility）の保証
本スキルをロードしたAIエージェントは、本セクション（Section 6〜7）に記述されたガバナンスルール、人間による承認（Sign-off）プロセス、不適合時の自己改善ループ、各レビュアーコメントの記述、および完全日本語化を**今後のセッションや他のプロジェクトにおいても自動的に再現し、忠実に実行しなければならない**。

## 9. スキルのライフサイクル管理とプロセス提案書の維持
カスタムスキルを運用するにあたり、スキルのライフサイクル（提案・作成、検証、バージョン管理、監査、自己改善、非推奨化/廃棄）を明確に定めた「組織カスタムスキルのライフサイクル管理プロセス提案書 (docs/skills_lifecycle_proposal.md)」を作成し、維持・更新しなければなりません。
エージェントは、スキルの追加・変更が発生した際、本提案書に記述された各ライフサイクルフェーズ（ADR作成、Evals自動検証、tagpr自動タグ付け、検査報告書作成、人間による承認等）に従って管理され、ガバナンスが機能していることを都度証明しなければなりません。

## 10. 他リポジトリ展開ガイドとポータビリティの保証
作成した組織カスタムスキル群および自己評価チェックリストの仕組みが、他のリポジトリへ容易に展開（ポータビリティ）できるように設計し、初心者やカスタマーサポート担当者が容易に参照できる「他リポジトリ展開ガイドおよび初期導入テンプレート (docs/deployment_guide_and_templates.md)」を作成・維持しなければなりません。
エージェントは、他プロジェクトへの移植を支援する際、本ガイドに記述された3ステップの手順、初期テンプレート（REQUIREMENTS.md, ADR）、およびトラブルシューティングFAQに従ってスムーズなオンボーディングを支援しなければなりません。
