# データベース設計 & 開発プロセス

## 1. 開発プロセス共通ルール

### 1.1 Single Source Of Truth (SSOT)
本プロジェクトにおけるスキーマ変更は、以下の「唯一の真実のソース」のフローに従います。
```
OpenAPI (api/openapi.yaml)
  ↓
ent (ent/schema/)
  ↓
Atlas Migration (ent/migrate/migrations/)
  ↓
Database
```

### 1.2 禁止事項
- **生成済みファイルの直接編集禁止**: 自動生成されたGoコードやSQLファイルを手動で修正してはなりません。
- **OpenAPIとentの同時編集禁止**: 片方の変更を行い、生成処理を完了した後に、次のステップの編集を行います。
- **migrationの手書き禁止**: マイグレーションSQLは必ず `Atlas CLI` を用いて自動生成し、手書きで追加・修正してはなりません。

### 1.3 命名規則
- **API**: `camelCase`
- **DB (テーブル名、カラム名等)**: `snake_case`
- **Go (構造体、フィールド名等)**: `PascalCase`

### 1.4 データベースコアルール
- **UUID利用**: すべてのテーブルの主キー（PK）はUUID（PostgreSQLの `uuid` 型）とします。
- **FK（外部キー）必須**: リレーションシップを表現する際は必ず外部キー制約を定義し、参照整合性を担保します。
- **N+1検出必須**: 関連するエンティティの取得時にN+1クエリが発生しないよう、事前ロード（Eager Loading）等を明示します。
- **Soft Delete（論理削除）は必要時のみ**: 原則として物理削除を前提とし、論理削除は法規制や業務要件上どうしても必要な場合のみ限定的に導入します。

## 2. 業務要件からDB設計への設計ルール

業務要件からデータベース設計を行う際は、以下のステップを**必ず**実行します。

1. **エンティティ抽出**: 業務要件に登場する主要な概念、オブジェクトをテーブル（エンティティ）として抽出します。
2. **リレーション抽出**: エンティティ間の関連（1対1、1対N、N対M）を定義します。
3. **ER図生成**: エンティティとリレーションの関係を視覚化します。
4. **ent schema生成**: Goの `entgo.io/ent` のスキーマ定義ファイルを記述します。
5. **Atlas migration生成**: スキーマ定義の変更差分からAtlas CLIを用いて migration ファイルを自動生成します。
6. **Index提案**: パフォーマンス向上のためのインデックス（特に複合インデックス）を提案します。
7. **FK提案**: 外部キー制約の定義を決定します。
8. **N+1検出**: 関連データのクエリ時にN+1が発生するリスクのある箇所を特定します。
9. **性能リスク検出**: 将来的なデータ増加による性能リスク（フルスキャン、巨大インデックス等）を検出します。

### 2.1 設計方針
- **UUID主キー**: PostgreSQL of `uuid` 型を主キーに使用します。
- **created_at / updated_at の追加**: すべてのテーブルに作成日時・更新日時を自動記録するカラムを追加します。
- **soft deleteは必要時のみ**: 原則は物理削除。論理削除は要件定義で明示された場合のみ。
- **edge明示**: `ent` スキーマにおいてリレーション（edge）を明確に定義し、双方向/単方向を適切に構成します。
- **PostgreSQL前提**: データベースはPostgreSQLを前提として設計します。
- **複合Indexを検討**: 検索クエリパターンに応じて、単一カラムだけでなく複合インデックス（Composite Index）の適用を積極的に検討します。

### 2.2 出力順
DB設計結果を出力する際は、必ず以下の順序で記載してください：
1. **ドメイン分析**: 業務要件の分析結果。
2. **エンティティ一覧**: 抽出したテーブルとカラムの一覧。
3. **ER図**: `mermaid` を利用したリレーションシップ図。
4. **ent schema**: `ent` スキーマ定義コード。
5. **Goコード**: アプリケーション側での使用例や自動生成されたGoの構造体イメージ。
6. **Atlas migration**: 生成されるマイグレーション手順および実行コマンド。
7. **SQL**: 生成されるDDL (Create/Alter Table等)。
8. **インデックス提案**: 作成すべきインデックスとその理由。
9. **性能上の懸念**: 大量データ時の負荷懸念。
10. **リスク**: 設計上のトレードオフやデータ整合性リスク（トランザクション負荷リスク含む）。

### 2.3 トランザクション設計・分析ルール
DB設計時にトランザクションの負荷・排他制御を最適化するため、以下の観点を**必ず**考慮・設計します。

#### 考慮事項 (7つのポイント)
1. **トランザクション境界**: データの整合性を保つ最小単位の範囲（境界）を定義します。
2. **読み取り/更新の分離**: 読み取り一貫性のみでよい処理（Replica参照可能）と、更新（Primary参照・更新）が必要な処理を分離します。
3. **ロック範囲**: テーブルロック、行ロック（SELECT FOR UPDATE等）の範囲を最小限に留めます。
4. **Isolation Level (分離レベル)**: 必要最低限の分離レベルを選択します（原則 Read Committed。 Serializable や Repeatable Read を使用する場合はその理由を明示）。
5. **Deadlockの可能性**: 同一トランザクション内での複数テーブル・複数行へのアクセス順序を一貫させ、デッドロックを防止します。
6. **長時間Transaction検出**: トランザクションが長時間オープンし続けるリスクを検知します。
7. **Retry必要性**: デッドロックやクエリタイムアウト時のアプリケーション側でのリトライ処理の必要性を評価します。

#### 実施原則 (5つの約束)
1. **Transactionを短く**: トランザクションの開始から終了（Commit/Rollback）までの時間を極限まで短縮します。
2. **外部API呼び出し禁止**: トランザクション内でHTTPリクエストなどの外部通信を行ってはなりません（外部通信のタイムアウトや遅延により、DBのロックや接続プールが枯渇するため）。
3. **sleep禁止**: トランザクション内でスレッドの待機（sleep等）を発生させてはなりません。
4. **大量更新はBatch化**: 大量のインサート・更新処理は、小さなバッチ（例: 100〜500件単位）に分割して段階的に実行します。
5. **SELECT FOR UPDATE乱用禁止**: 悲観的ロックは本当にデータ競合が頻発する箇所以外では使用せず、楽観的ロック（バージョン管理カラム等）を検討します。

#### 他のSkillsとの連携（スキーマ見直し・コード修正）
- **OpenAPI設計（openapi-design）との連携**: トランザクション境界を短くするため、APIの設計そのものを「非同期型（受付のみ行いバックグラウンドで処理）」や「複数APIへの分割」に変更できないか検討し、API定義の見直しを提案します。
- **Go実装（golang-design）との連携**: トランザクションブロック内で外部呼び出しや余分な処理が行われないよう、関数の責務分割や、リトライ・バッチ処理の実装コード修正を指示・反映します。

## 3. 実装・運用ガイドライン (DBAベストプラクティス)

### 3.1 SQLファースト方針 (No-ORM)
- **生のSQLの推奨**: パフォーマンスの不確実性と過剰な抽象化を避けるため、複雑なORMは採用せず、生SQLまたはそれに近いクエリ記述ツールを使用し、開発者が制御可能な明示的な SQL を書く。
- **型安全な SQL クエリ**: スキャン漏れやパラメータのマッピングミスを防ぐため、名前付きプレースホルダーを利用し、動的SQLの生成は最小限にする。

### 3.2 コネクションプールのチューニング
データベースへの過剰な接続やリソースリークを防ぐため、必ず最大同時接続数、アイドル時接続数、および最大接続生存期間を環境に合わせて最適に設計・設定する。
- **接続数上限の算出**:
  - `MaxOpenConns`（最大同時接続数）は、データベースサーバーが許容できる最大接続数（`max_connections`）を、アプリケーションのインスタンス（レプリカ）数やバッチ処理スレッド数で割った値よりも小さく設定する。
- **アイドル接続数の最適化**:
  - `MaxIdleConns`（アイドル時の最大接続数）は、極力 `MaxOpenConns` と同値に設定することを推奨する。これにより、接続の切断・再作成のスパイクを最小化し、TCPハンドシェイクのオーバーヘッドを削減する。
- **最大生存期間（Lifetime）**:
  - `ConnMaxLifetime`（接続の最大生存期間）は数分〜数十分程度に設定し、長時間の接続によるメモリリークやロードバランサによる強制切断からリソースを保護する。

### 3.3 トランザクションと分離レベルの管理
- **Context の伝搬**: トランザクション処理には必ずコンテキスト（タイムアウト・キャンセル情報）を伝搬させる。
- **確実なロールバック処理**: トランザクション開始直後にロールバックを遅延実行（defer）し、コミット成功時のみフラグを反転させることで、例外発生時のロールバック漏れを防ぐ。
  ```go
  tx, err := db.BeginTx(ctx, &sql.TxOptions{Isolation: sql.LevelReadCommitted})
  if err != nil {
      return err
  }
  defer tx.Rollback()
  ```
- **分離レベルの選択**:
  - デフォルトは `Read Committed`。
  - 厳密な整合性が必要な処理では、`Serializable` などを指定し、アプリ側でのリトライロジックを実装する。

### 3.4 データベースマイグレーション
- **スキーマ管理のコード化**: スキーマ変更は手動で行わず、必ずバージョン管理マイグレーションツールを利用し、SQLファイル（Up/Down）としてリポジトリで管理する。
- **Atlas CLIによるバージョン管理マイグレーション (Versioned Migration)**:
  - マイグレーションツールとして `Atlas CLI` (atlasgo.io) を標準採用する。
  - `ent` スキーマ（または生SQL）から現在のスキーマ状態と変更後のスキーマ状態を Atlas で比較し、自動的にマイグレーションSQLファイル（`migrations/` ディレクトリ）を生成・バージョン管理する。
  - DDLの変更時は、適用前に `atlas migrate lint` などの静的解析を実行し、データ消失（破壊的変更）やパフォーマンス低下を伴うロック等が発生しないかを検証する。
- **マイグレーションの自動実行**: マイグレーションはデプロイパイプラインの初期フェーズで自動実行されるようにする。
- **後方互換性の維持**: インフラの稼働率を下げることなくデプロイを行うため、マイグレーションは常に「後方互換性」を考慮して設計する。

### 3.5 クエリ最適化と監視
- **EXPLAIN による実行計画評価**: インデックスが効いていないフルスキャンを避けるため、クエリチューニング時はデータベース上で `EXPLAIN` や `EXPLAIN ANALYZE` を実行し実行計画を評価する。
- **スロークエリ監視**: 定義した閾値（例: 200ms）を超える SQL クエリは、警告レベルでログ出力し、実行時間と引数を記録し監視する。
- **メトリクス監視**: プロダクション環境では、現在の接続プール状態をメトリクスとして公開し、リソースの飽和度を監視する。

### 3.6 Ent ORM (entgo.io) の設計・運用指針
- **接続プールの最適設定**:
  - `ent.Open` 時に返される `*ent.Client` にて、内部ドライバー（`sql.DB`）に対して `database-design:3.2` で規定された接続プール設定を必ず適用すること。
- **型安全スキーマの定義とDB制約**:
  - スキーマ定義において、適切な一意性制約（`Unique()`）やインデックス（`Indexes()`）を明示的に定義し、Goの型安全性とDB側の整合性を両立させること。
- **トランザクション管理と二重登録防止**:
  - 書き込み操作では、必ず `client.Tx(ctx)` を用いてトランザクションを明示的に制御し、`defer rollback` パターンを実装すること。
  - 同期再送時の二重インサートを防止するため、`Upsert` (ON CONFLICT DO UPDATE/NOTHING) をスキーマのユニークキーに対して正しく構成すること。
- **自動マイグレーションの運用監視**:
  - ローカル開発環境では `client.Schema.Create(ctx)` による自動マイグレーションを許容するが、本番環境のデプロイにおいては、`client.Schema.Create` は使用せず、`ent` のスキーマ定義を Atlas に食わせることで生成されたマイグレーション SQL ファイルを使用し、Atlas CLI (またはそれに準ずるマイグレーション適用プログラム) 経由で適用すること。
  - マイグレーションファイルは CI や本番デプロイ時に事前に DDL 解析とレビュー（手動承認）を経て適用されること。
