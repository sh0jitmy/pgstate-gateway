# トランザクション・プロファイリング & チューニングプロセス

## 1. 開発プロセス共通ルール

### 1.1 Single Source Of Truth (SSOT)
本プロジェクトにおけるスキーマ変更は、以下の「唯一の真実のソース」のフローに従います。
```
OpenAPI (api/openapi.yaml)
  ↓
ent (ent/schema/)
  ↓
Atlas Migration (ent/migrate/migrations/)
  ↓
Database
```

### 1.2 禁止事項
- **生成済みファイルの直接編集禁止**: 自動生成されたGoコードやSQLファイルを手動で修正してはなりません。
- **OpenAPIとentの同時編集禁止**: 片方の変更を行い、生成処理を完了した後に、次のステップの編集を行います。
- **migrationの手書き禁止**: マイグレーションSQLは必ず `Atlas CLI` を用いて自動生成し、手書きで追加・修正してはなりません。

### 1.3 命名規則
- **API**: `camelCase`
- **DB (テーブル名、カラム名等)**: `snake_case`
- **Go (構造体、フィールド名等)**: `PascalCase`

### 1.4 データベースコアルール
- **UUID利用**: すべてのテーブルの主キー（PK）はUUID（PostgreSQLの `uuid` 型）とします。
- **FK（外部キー）必須**: リレーションシップを表現する際は必ず外部キー制約を定義し、参照整合性を担保します。
- **N+1検出必須**: 関連するエンティティの取得時にN+1クエリが発生しないよう、事前ロード（Eager Loading）等を明示します。
- **Soft Delete（論理削除）は必要時のみ**: 原則として物理削除を前提とし、論理削除は法規制や業務要件上どうしても必要な場合のみ限定的に導入します。

## 2. トランザクション・プロファイリング手順

長時間トランザクション（Long-Running Transaction: LTT）やデッドロック、インデックス未適用によるスロークエリを検知・分析する際は、以下のステップを**必ず**実行します。

### 2.1 EXPLAIN / EXPLAIN ANALYZE によるクエリプラン解析
- **フルスキャン (Seq Scan) の検出**: `WHERE` 句でインデックスが使われず、テーブル全体を走査しているクエリを特定します。
- **コストと実行時間の評価**: 最も実行コストが高いノード（Nest Loop, Hash Join, Sort 等）を特定し、ボトルネックを明確にします。
- **ロック競合の確認**: PostgreSQLの `pg_locks` ビューや、ロック獲得待ち（`LockAcquisitionException` やログの `exclusive lock` 待機）を調査し、競合の発生源を特定します。

### 2.2 OpenTelemetry トレースによるトランザクション分析（golang-observabilityとの連携）
- **トランザクション全体の生存時間 (Tx Duration) の計測**:
  - トランザクション開始（`db.BeginTx`）から、コミット/ロールバック終了までのスパンを計測します。
- **ボトルネック特定**:
  - トランザクションスパン内に含まれる子スパン（各SQLクエリの実行スパン、およびそれに挟まれたアプリケーションロジックのスパン）を比較します。
  - **ブロッキング要素の特定**: トランザクションの生存時間に対して、各クエリ実行時間の合計値が著しく低い場合、クエリ以外の処理（外部API通信や `time.Sleep` などの同期的なブロッキング処理）が内部に紛れ込んでいることを特定します。

### 2.3 pprof を用いたアプリケーション内ロック競合分析
- **mutex / block プロファイルの実行**:
  - トランザクション処理中にGoルーチンがロック取得（`sync.Mutex` やデータベースコネクションプール等）で待機している時間を分析します。
- **CPU / メモリプロファイルの活用**:
  - 大量データのバッチ処理時に発生する、不要なメモリ割り当て（GC負荷）やCPU計算負荷の高いホットスポットを特定します。

### 2.4 デッドロック分析
- **ログ情報の抽出**:
  - データベースログから `deadlock detected` エラーとその前後で実行されていたトランザクションのクエリ（Lock Graph）を取得します。
- **アクセス順序の不整合の特定**:
  - トランザクションAとトランザクションBが、複数のテーブルまたは行に対して、互いに異なる順序でロック（または更新）を掛けようとした箇所を特定します。

## 3. 分析結果と改善・修正提案の出力ルール

プロファイリングを完了した後は、必ず以下の構造に従って、スキーマ見直しおよびコード修正の提案を日本語で出力してください。

### 3.1 分析サマリー
- 長時間トランザクションまたはデッドロックの発生原因と、影響を受けているエンドポイント・テーブルを整理します。

### 3.2 スキーマの見直し提案（database-designと連携）
- **インデックスの適用・見直し**: 複合インデックスの提案、部分インデックス（Partial Index）の適用。
- **ロック範囲の縮小**: 更新処理の競合を減らすためのスキーマの非正規化、または楽観的排他制御用カラム（`version` 等）の追加。

### 3.3 Goコードの修正案（golang-designと連携）
- **Txブロックの短縮**: トランザクション外に逃がすべき処理（外部API呼び出し、ログ出力、非同期通知等）のコード差分（diff）。
- **バッチ処理化**: ループ内の単一インサート処理を、チャンク分割された一括更新（Batch/Bulk）に変更するコード差分（diff）。
- **リトライ処理の追加**: シリアライズエラー時に自動で指数バックオフを伴うリトライを行うためのヘルパー関数適用のコード差分（diff）。
