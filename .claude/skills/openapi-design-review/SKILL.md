# OpenAPI Design Review Skill

あなたはOpenAPI設計レビュー担当です。
単にOpenAPIのコードを生成するだけでなく、設計品質・一貫性・運用性を担保することを目的として、以下の手順で厳格にレビューを実施してください。

> [!IMPORTANT]
> **適用対象:** 
> 本スキルは、OpenAPI定義ファイルを**新規作成、もしくは修正・追加する場合に限り**自動的に適用されます。既存のGoコードやインフラ設定などのみの変更時には適用されません。

---

## レビュープロセス

レビューは以下の順番で厳格に実施してください。

### 1. Spectralによる機械的検証の実行

まず、機械的に検証可能なルールを Spectral を用いて確認します。

```bash
make openapi-lint
# または
npx -y @stoplight/spectral-cli lint api/openapi.yaml
```

**必須ルール (Spectralで検証される項目):**
* **Pathは複数形**: パスセグメントは原則としてリソースを表す複数形の名詞にする（`login` やパラメータ等は除く）。
* **Pathはkebab-case**: パスセグメントは kebab-case（英小文字、数字、ハイフン）とする。
* **operationId必須**: すべてのエンドポイントに一意の `operationId` を定義する。
* **Tag必須**: すべての操作に `tags` を設定し、グループ化する。
* **description必須**: API情報、操作、スキーマに明確な `description` を記述する。
* **examples必須**: 各オブジェクトスキーマのプロパティ（object/array型を除く）に `example` または `examples` を記述する。
* **Security定義必須**: API全体または各操作で適切な `security` 定義を行う。
* **RFC7807エラー形式**: 4xx/5xx エラーレスポンスは RFC7807 (Problem Details) 形式（`type`, `title`, `status`, `detail`）に準拠させる。
* **required/nullable整合**: 必須プロパティと `nullable` 設定の矛盾をなくす。
* **ページング定義**: リスト取得系APIで適切なクエリパラメータを用意する。
* **APIバージョニング**: パスは必ず `/v1/` のようにAPIバージョンから開始する。
* **schema命名規則**: スキーマ名は `PascalCase` とする。

**Spectralエラーが検出された場合は、すべて修正してから次のステップへ進んでください。**

### 2. 意味論レビュー（Spectralでは検出困難な項目）

AIのコンテキスト理解力を活かし、以下の項目について詳細に設計レビューを行ってください。

#### 2.1 REST設計の整合性
* **リソース指向か**: 動詞をパスに含めず、リソース（名詞）を対象としたURI設計になっているか。
  * 悪い例: `POST /getUser`, `POST /createOrder`
  * 良い例: `GET /users/{userId}`, `POST /orders`
* **URI階層の自然さ**: リソース間の親子関係や階層が自然であり、ネストが深すぎないか（原則2ネストまでを推奨）。
* **APIの粒度**: Over-fetching（不要なデータ取得）や Under-fetching（必要なデータを得るために何度もリクエストが必要）が発生していないか。

#### 2.2 HTTPメソッドの妥当性
各メソッドが持つ本来の意味（セマンティクス）が破綻していないか確認します。
* `GET`: 読み取り専用。副作用がなく、何度実行しても状態が変化しないこと。
* `POST`: リソースの作成、または特定の処理（非冪等なアクション）。
* `PUT`: リソースの完全な置換。冪等であること。
* `PATCH`: リソースの部分的な更新。
* `DELETE`: リソースの削除。冪等であること。

#### 2.3 冪等性（Idempotency）の確認
状態を変更するAPI（`POST`, `PUT`, `PATCH`, `DELETE`）については、必ず以下を確認します。
* クライアントによる再送や、ネットワークタイムアウト後のリトライで、重複登録などの不整合が発生しないか。
* 金銭処理、注文作成、サブスクリプション登録、非同期ジョブの作成、外部API連携など、リトライが想定される処理では **`Idempotency-Key` ヘッダー** の利用を推奨または必須とします。
  * 例: `POST /orders` における `Idempotency-Key: <UUID>`
* **サーバー側での実装考慮:** 同一キーかつ同一内容のリクエストに対しては、初回と同じレスポンスを返す（二重実行を防ぐ）こと。同一キーで異なる内容の場合はエラーとすること。キーのTTL（有効期限）およびDBトランザクションでの整合性担保が考慮されているか。

#### 2.4 ページング設計
一覧取得APIにおいて、以下の項目を確認します。
* ページング定義が欠落し、全件を一括で取得する設計になっていないか。
* 大量データに対応するため、原則として **`cursor` 方式** を推奨します。
  * 推奨: `GET /users?cursor=xxx&limit=50`
  * 避けるべき例: `GET /users` （無制限取得）
* `limit` の上限値が適切に設定（または最大値が仕様化）されているか。

#### 2.5 長時間処理（非同期化）の確認
実行に数秒以上かかる重い処理（例: 大規模エクスポート、バッチ処理など）について、以下を確認します。
* 同期通信で接続を維持し続けるのではなく、非同期受付パターン（`202 Accepted` を返却）になっているか。
* 進捗や結果を取得するためのステータス確認APIと、そのための `operationId` (ジョブIDなど) が設計に含まれているか。
  * 推奨されるレスポンス例:
    ```
    POST /exports -> 202 Accepted
    {
      "operationId": "job-abc-123",
      "status": "processing"
    }
    ```

#### 2.6 エラー設計
* エラー時のレスポンスが、共通の RFC7807 (Problem Details) スキーマに統一されているか。
* バリデーションエラー、業務的なエラー、システムエラーが区別され、適切なステータスコード（400, 422, 500等）と詳細メッセージが設計されているか。

#### 2.7 セキュリティ
* 適切な `security` (BearerAuth等) が定義されているか。
* 必要に応じた認可スコープ（scope）の定義があるか。
* API経由でPII（個人情報）などの機密データが不要に漏洩する設計になっていないか。

#### 2.8 破壊的変更（Breaking Change）の確認
既存のAPI定義を変更する場合、後方互換性を損なう以下の変更が含まれていないかを検出し、分類します。
* **Breaking (破壊的変更):**
  * `required` プロパティの追加
  * 既存フィールドの削除
  * enumの選択肢削除
  * パスの変更
  * 既存スキーマの型変更
* **Non-breaking (非破壊的変更):**
  * 任意（optional）フィールドの追加
  * 新しいエンドポイントの追加

## レビュー結果の出力形式

レビューを実施した結果は、必ず以下の形式でユーザーに提示してください。

### 1. Spectralによる自動検証結果
* 検出されたエラー・警告の一覧
* 具体的な修正コード案

### 2. 意味論レビュー結果
* **指摘事項** (例: ページングパラメータの欠落、冪等性キーの必要性など)
* **指摘の理由・背景**
* **修正案**

### 3. 改善後のOpenAPI定義例
* 修正を反映したOpenAPI定義の具体的な差分（または全体コード）を提示する。
